Agentic RAG 企业知识库系统 – v0.1 需求文档

虚拟环境路径：E:\anaconda3\envs\graph

1. 版本目标
将现有桩实现（框架+测试）升级为可工作的 MVP，能够：

加载 EDITH 数据集的真实文档块和索引

调用 LLM（dsv4，DEEPSEEK_API_KEY通过环境变量获取）完成意图分类、多跳规划、充分性评估、答案生成

通过混合检索（FAISS + BM25）返回真实文档片段

在命令行界面回答用户问题，并输出完整的推理轨迹

里程碑：用户运行 python main.py "What was Veracier's R&D spending in 2022?" 后能得到基于真实文档的合理答案（非占位符）。

3. 任务分解（按优先级 P0 必须完成）
P0-1: 实现真实检索器（retriever.py）
当前问题：retrieve 节点返回假 chunks。
要求：

在 retriever.py 中实现 HybridRetriever 类。

构造函数加载：

chunks.jsonl（由 data_prepare.py 生成）

faiss.index（向量索引）

bm25_index.pkl（BM25 索引）

提供 search(query: str, top_k: int = 5) -> List[RetrievedChunk] 方法：

使用 sentence_transformers 编码 query 为向量（模型路径从 config.EMBEDDING_MODEL 读取）

执行 FAISS 检索（index.search），获得向量得分和索引位置

执行 BM25 检索（bm25.get_scores），获得 BM25 得分

使用 RRF 融合（k = config.RRF_K）计算最终分数

返回 RetrievedChunk 对象列表，按分数降序排序

要求确定性：固定随机种子（np.random.seed(42)）。

如果索引文件不存在，抛出 FileNotFoundError 并提示用户运行 data_prepare.py。

验收标准：

运行 python -c "from retriever import HybridRetriever; r=HybridRetriever(); print(r.search('R&D spending', 3))" 能返回真实 chunks（非空，有文本）。

相同查询连续两次返回的 chunk_id 顺序一致。

P0-2: 实现 LLM 调用封装（llm_client.py，新增文件）
需求：

创建一个新模块 llm_client.py。

实现 get_llm_response(prompt: str, system_prompt: str = None, temperature: float = 0.0) -> str 函数。

支持 OpenAI API（openai.OpenAI）和 Anthropic API（可选，如果环境变量有 ANTHROPIC_API_KEY）。

读取配置：

config.LLM_PROVIDER（openai 或 anthropic）

config.LLM_MODEL（默认 gpt-4o-mini）

config.LLM_TEMPERATURE（默认 0.0）

实现重试逻辑：最多重试 3 次，指数退避（初始 1s），捕获 RateLimitError 和 APITimeoutError。

对返回的内容做 .strip()，去除首尾空白。

验收标准：

单元测试（可手动）调用 get_llm_response("Say 'hello'") 返回 "hello"（或近似）。

没有 API Key 时给出明确错误提示。

P0-3: 替换工作流节点的桩实现（workflow.py）
逐个修改节点函数，将硬编码返回值替换为真实的 LLM 调用或逻辑。每个节点必须遵守原状态定义。

节点 classify_intent
输入：state["question"]

输出：intent 字段（"simple", "multi_hop", "unclear"）

实现：

构造提示词（参考 MVP 文档中的模板，确保 LLM 只返回一个单词）。

调用 llm_client.get_llm_response。

如果返回值不是三个合法值之一，默认设置为 "simple" 并记录警告到 reasoning_trace。

验收：对模糊问题（如“Tell me about the company”）返回 "unclear"。

节点 plan_sub_questions
输入：state["question"]

输出：sub_questions 列表（至少两个元素）

实现：

提示词：“请将以下问题分解为 2-3 个独立的子问题，每个子问题可以单独通过检索回答。只输出子问题列表，每行一个，不要有序号或额外文字。”

调用 LLM，按换行分割结果，清理空行。

如果 LLM 返回少于 2 个子问题，使用后备策略：将原问题重复两次（简单拆分）。

验收：对“哪个部门负责碳中和计划？”应输出类似 ["What is Veracier's carbon neutrality target year?", "Which department is responsible for that target?"]

节点 check_sufficiency
输入：当前问题（可能是 state["sub_questions"][current_sub_idx] 或 state["question"]），state["accumulated_chunks"]

输出：information_sufficient (bool) 和 missing_information (str or None)

实现：

将累积的 chunks 格式化为文本（截断过长内容，每个 chunk 最多 500 字符）。

使用提示词模板（见 MVP 文档），要求 LLM 输出 SUFFICIENT 或 INSUFFICIENT: <missing info>。

正则解析输出，如果无法解析，默认 sufficient=True（保守策略）。

验收：当提供的 chunks 明显缺少关键数字时，应返回 False 并描述缺失信息。

节点 refine_query
输入：state["question"] 和 state["missing_information"]

输出：new_query 字符串

实现：

提示词：“原始问题：{question}。缺失的信息：{missing_information}。请生成一个更精确的检索查询（一句话），以便找到缺失的信息。只输出查询语句。”

调用 LLM，返回清洗后的字符串。

如果 LLM 调用失败，回退为在原查询前加 "refined: " 前缀。

验收：原问题“2022年研发支出”，missing_info“缺少具体数字”，应生成类似 "2022 R&D spending amount in euros"。

节点 generate_answer
输入：问题（原问题）和 state["accumulated_chunks"]

输出：final_answer 字符串，必须包含引用（chunk_id）

实现：

提示词：“基于以下文档片段回答用户问题。不要编造信息。如果文档中没有答案，明确说‘没有找到相关信息’。在每个引用句后附上来源 ID（如 [doc_5]）。\n\n文档片段：\n{chunks}\n\n问题：{question}\n\n答案：”

调用 LLM，获取答案。

如果答案中未出现任何 [doc_xxx] 形式的引用，自动在末尾添加 [来源：未明确]。

验收：答案中至少包含一个 chunk_id。

节点 retrieve (已有框架，需填充真实检索器)
当前状态：桩实现，返回假 chunk。

修改为：

从 state 中获取当前查询（如果是 simple 路线则是 state["question"]，如果是 multi_hop 则是 state["sub_questions"][current_sub_idx]）。

调用 HybridRetriever.search(query, top_k=config.TOP_K)。

将返回的 RetrievedChunk 列表追加到 state["accumulated_chunks"]（去重：按 chunk_id）。

更新 state["reasoning_trace"] 记录检索到的 chunk 数量。

注意：需要确保 HybridRetriever 是单例（避免重复加载索引），可以在模块级初始化。

验收：在 main.py 中运行一个简单问题，观察 state["accumulated_chunks"] 不为空，且内容与问题相关。

P0-4: 修复工作流路由和状态转换（workflow.py）
当前路由函数已存在，但需要与真实节点行为配合。检查以下路由条件：

当 check_sufficiency 返回 False 且 retry_count < max_retries 时，应进入 refine_query 然后回到 retrieve。

当 retry_count 达到上限时，直接进入 generate_answer，并在最终答案中声明“信息不足”。

确保 multi_hop 路径中，所有子问题完成后，最后一步用 accumulated_chunks 回答原问题。

验收：运行一个需要两次检索的问题（例如 “What is the name of the subsidiary that achieved the highest revenue in 2021?”），观察工作流是否完成至少两次检索循环。

P0-5: 增加端到端集成测试（test_e2e.py，新增文件）
创建新测试文件，要求：

使用 EDITH 提供的 ANSWER_KEY.json（如果存在）或手动选取 5 个简单问题。

对每个问题，调用 main.py 的 run_workflow 函数（或直接调用 graph.invoke）。

验证：

最终答案字符串非空且长度 > 10。

答案中不包含“占位”“placeholder”等词。

至少检索到 1 个 chunk（retrieved_sources 非空）。

不要求答案完全匹配 ground truth，但必须与问题相关。

验收：运行 python test_e2e.py 输出“5/5 通过”或给出具体失败原因。

P0-6: 更新 data_prepare.py 确保可生成完整索引
当前 data_prepare.py 是框架，需确保：

能正确读取 EDITH 目录下的所有 PDF（或指定 --limit）。

使用 fitz (PyMuPDF) 提取文本，对于扫描件调用 OCR（pytesseract）。

分块策略：按段落分割，每个块最大 chunk_size 字符，重叠 chunk_overlap。

生成 chunks.jsonl、faiss.index、bm25_index.pkl。

如果 OCR 失败，跳过该 PDF 并记录日志。

验收：运行 python data_prepare.py --limit 10 --skip-ocr 能在 2 分钟内生成有效的三个文件。

5. 验收标准（v0.1 完成的定义）
✅ 运行 python data_prepare.py --limit 200 生成索引无误。

✅ 运行 python main.py --interactive 能正常对话，至少回答一个简单问题（如 “What is Veracier Industries?”）且答案内容合理。

✅ 所有 LLM 调用节点均使用真实 API，无硬编码返回值。

✅ 工作流在遇到信息不足时能触发 refine_query 重试（最多 3 次）。

✅ 输出 JSON 包含 reasoning_trace 字段，记录至少 3 步决策过程。

6. 风险与缓解措施
风险	缓解措施
LLM API 费用过高	设置 temperature=0, 使用便宜的 gpt-4o-mini，在测试时使用小样本。
OCR 处理过慢	默认 --skip-ocr，仅对需要 OCR 的 PDF 单独处理；v0.1 中可跳过扫描件，只使用可搜索 PDF。
检索召回率低	先不做深度优化；如果太低，在 retrieve 中临时增加 top_k=10。
LLM 返回格式不符合预期	每个节点增加正则 fallback，保证工作流不中断。

三、使用方法
========================================================================

3.1 环境准备
-----------
# 1. 激活虚拟环境
E:\anaconda3\envs\graph\Scripts\activate

# 2. 安装依赖
pip install -r requirements_v0_1.txt

# 3. 设置 API Key（必需）
# Windows PowerShell:
$env:DEEPSEEK_API_KEY = "sk-xxxxxxxxxxxxxxxx"
# 或 OpenAI:
$env:OPENAI_API_KEY = "sk-xxxxxxxxxxxxxxxx"

# 4. 设置正确的模型名（DeepSeek 用户必须）
$env:RAG_LLM_MODEL = "deepseek-v4-pro"

# 5. （可选）覆盖其他配置
$env:RAG_TOP_K = "5"
$env:RAG_MAX_RETRIES = "3"
$env:RAG_CHUNK_SIZE = "512"

3.2 生成索引
-----------
# 快速测试（处理 10 个 PDF，跳过 OCR）
python data_prepare.py --limit 10 --skip-ocr

# 完整索引（处理前 200 个 PDF）
python data_prepare.py --limit 200 --skip-ocr

# 完整索引 + OCR
python data_prepare.py --limit 200

# 使用本地已缓存的 embedding 模型
python data_prepare.py --limit 50 --embedding-model BAAI/bge-small-zh-v1.5

# 输出文件：
#   chunks.jsonl      — 文档块（JSONL格式）
#   faiss.index       — FAISS 向量索引
#   bm25_index.pkl    — BM25 稀疏索引

3.3 运行问答
-----------
# 单个问题
python main.py "What is Veracier Industries?"

# 交互模式
python main.py --interactive

# 交互模式（详细日志）
python main.py --interactive --verbose

# 批量问答（从文件读取）
python main.py --file questions.txt

# 输出 JSON 到文件
python main.py "What was the R&D spending?" --output result.json

# 交互模式命令：
#   /help    — 显示帮助
#   /quit    — 退出
#   /status  — 显示索引和配置状态
#   /graph   — 显示工作流拓扑

3.4 运行测试
-----------
# 结构性测试
python test_mvp.py --verbose

# 端到端测试
RAG_LLM_MODEL=deepseek-v4-pro python test_e2e.py --verbose --limit 5

# 注意：e2e 测试的"来源非空"检查需要先生成索引
# 无索引时该检查会自动跳过

DeepSeek 用户注意：
  必须设置 RAG_LLM_MODEL=deepseek-v4-pro（或 deepseek-v4-flash），
  因为默认值 gpt-4o-mini 不被 DeepSeek API 接受。
  API Key 设置 DEEPSEEK_API_KEY 即可自动识别并使用 DeepSeek 端点。


========================================================================
五、工作流拓扑
========================================================================

START → classify_intent
    ├── unclear   → ask_clarification → END
    ├── simple    → retrieve → check_sufficiency
    └── multi_hop → plan_sub_questions → retrieve → check_sufficiency

check_sufficiency →
    ├── sufficient + done      → generate_answer → END
    ├── sufficient + next_sub  → retrieve (下一子问题)
    ├── insufficient + retry<3 → refine_query → retrieve
    └── insufficient + retry=3 → generate_answer (强制，附免责声明)


========================================================================
六、关键设计决策
========================================================================

1. HybridRetriever 单例模式
   - retriever.py 与 workflow.py 存在循环导入：
     retriever 需要 workflow.RetrievedChunk
     workflow 需要 retriever.HybridRetriever
   - 解决：workflow.py 中使用 _get_retriever() 懒加载函数

2. retry_count 递增时机
   - 必须在搜索尝试之前递增（在 try/except 之外）
   - 原因：搜索失败（如索引缺失）时仍应计数，否则会无限循环

3. Fallback 策略
   - classify_intent: 异常时默认 "simple"
   - plan_sub_questions: 子问题<2个时拆分为2部分
   - check_sufficiency: 异常时默认 sufficient=True（保守）
   - refine_query: 异常时加 "refined:" 前缀
   - generate_answer: 异常时输出错误信息+已检索文档列表
   - retrieve: 索引缺失时记录错误不崩溃

4. 路由函数中的状态变更
   - route_by_sufficiency 在 multi_hop 场景下会修改
     current_sub_idx 和 active_query
   - LangGraph 传递 state 引用，变更会持久化
   - generate_answer 始终使用 state.question（原问题）


========================================================================
七、已知限制（v0.1）
========================================================================

1. HuggingFace 模型下载
   - BAAI/bge-base-en-v1.5 需要联网下载到缓存
   - 若无法联网，使用 --embedding-model 指定本地可用模型
   - 本地已缓存 bge-small-zh-v1.5（中文优化，512维）

2. OCR 处理
   - 默认 --skip-ocr 跳过扫描件（约25%的PDF）
   - OCR 需要系统安装 Tesseract + poppler
   - Windows 用户需额外配置

3. 检索质量
   - 未使用 Cross-encoder 重排序
   - 未使用语义缓存
   - 多语言检索未做优化

4. API 费用
   - 每次问答调用约 5-15 次 LLM API（取决于重试次数）
   - 建议使用 deepseek-v4-pro（性价比高）

2.1 问题场景：重试循环死循环（P0 严重）
----------------------------------------------------------------------

用户输入问题：
  "How many subsidiaries does Veracier have, and in which countries
   are they located?"

实际推理轨迹：
  [classify_intent] intent: simple → simple          ← 错误！应为 multi_hop
  [retrieve] top_k=5  → 5 chunks
  [check_sufficiency] 信息不足 → retry 0/3
  [refine_query] → 新查询词
  [retrieve] top_k=5  → 10 chunks
  [check_sufficiency] 信息不足 → retry 1/3
  ...重复...
  [check_sufficiency] 已达最大重试 → 强制生成
  [generate_answer] Answer: 97 chars              ← 无用输出

根因分析：
  A. Intent 分类错误："how many X AND which countries" 是典型的 compound
     question，但被 LLM 判为 "simple" → 跳过了多跳检索流程。

  B. check_sufficiency 过度严格：LLM judge 的 system prompt 是
     "Be critical — if key facts or specific numbers are missing,
      judge as INSUFFICIENT"。
     对于"列出全部子公司+国家"这种问题，任何 chunk 集合都不可能包含
     完整的穷举列表 → LLM judge 永远判 INSUFFICIENT → 必然耗尽3次重试。

  C. 重试不扩大检索范围：每次重试都用 top_k=5，新 chunk 去重后净增极少
     (5→10→13→16)，4次检索才覆盖16个独特chunk。

修复方案：
  ✅ A. classify_intent 改进 (workflow.py:244-266)
     新增规则："and" 连接的多个独立子问题 → multi_hop
     System prompt 强调：计数+列举的组合问题几乎总是 multi_hop

  ✅ B. check_sufficiency 渐进式宽松 (workflow.py:674-730)
     retry 0: 严格模式 — 关键事实缺失 → INSUFFICIENT
     retry 1: 温和模式 — 有部分信息能给出有用回答 → SUFFICIENT
     retry 2+: 自动通过 — 不调用 LLM，直接设 sufficient=True
     这是核心修复：避免 LLM judge 对"穷举型"问题无限判 INSUFFICIENT

  ✅ C. top_k 随重试递增 (workflow.py:1353-1366)
     Cycle 0 (首次): top_k = 5
     Cycle 1 (重试1): top_k = 10  (×2)
     Cycle 2 (重试2): top_k = 15  (×3)
     扩大搜索范围，更快覆盖不重复的 chunk

2.3 问题场景：multi-hop retry 耗尽后跳过后续子问题（P0 严重）
----------------------------------------------------------------------

用户输入问题：
  "Compare Veracier's revenue growth between 2021 and 2022 across its
   main business segments"

实际行为：
  [classify_intent] intent: multi_hop → 拆解为 2 个子问题
  [retrieve] 子问题1 (2021 revenue) → 信息不足 → retry 0/3
  [refine_query] → 新查询词 → retrieve → 仍不足 → retry 1/3
  [refine_query] → retrieve → 仍不足 → retry 2/3
  [refine_query] → retrieve → 仍不足 → 已达最大重试
  然后直接跳到 generate_answer，**子问题2 (2022 revenue) 完全没有被检索**

根因分析：
  run_workflow_streaming() 中外层循环为：
    max_cycles = rt.max_retries + 1  # = 4
    for cycle_idx in range(max_cycles):
  第一个子问题消耗了全部 4 个 cycle（1 初始 + 3 retry），cycle_idx=4
  时直接退出 for 循环，不再为第二个子问题执行检索。

  本质：max_cycles 是所有子问题共享的硬上限，而非每个子问题独立计算。

修复方案：
  ✅ 将外层循环从 `for cycle_idx in range(max_cycles)` 改为 `while True`
     + 内部手动管理 cycle 计数 (workflow.py:1354)
  ✅ 在切换到下一个子问题时重置计数器：
     state.retry_count = 0  (workflow.py:1457)
  ✅ 子问题推进逻辑在 retry 耗尽分支中：
     当前子问题耗尽 → 检查是否还有剩余子问题 → 有则推进并 continue
     继续 while True 循环为该子问题服务

验证结果：
  ✅ 已通过测试：multi-hop 问题的每个子问题现在都能独立获得完整的
     retry 配额（各自最多 3 次 retry），不再出现子问题被跳过的情况。

3.4 切分策略升级：缩写感知的句子级分块
----------------------------------------------------------------------

问题回顾（v0.2 切分策略）：
  原始策略按双换行符分段，再按 ". "（句号+空格）切分句子。对于包含
  缩写的欧洲企业文档（如 "M. Dupont S.A."、"GmbH"、"Dr. Rossi"），
  缩写后的句号会被误判为句子边界，导致：
    - "Precis-Tec S.A." → 切断为 "Precis-Tec S." 和 "A."
    - "M. Dupont" → 切断为 "M." 和 "Dupont"
    - 切分碎片进入不同 chunk，语义完整性被破坏

改进方案（v0.3.1 句子优先 + 缩写感知）：

  ✅ 策略1（优先）：NLTK punkt 分词器
    - 使用预训练的 punkt 模型，原生处理缩写（Mr./Dr./S.A./GmbH）
    - 支持 5 种语言：en/fr/de/it/es
    - 自动下载缓存，离线可用

  ✅ 策略2（回退）：正则三阶段分词器
    Pass 0 — 段落级分割：
      按双换行符 (\n\s*\n) 拆分为段落，保留段落边界语义

    Pass 1 — 句子边界检测：
      正则 (?<=[.!?。！？])[\s\n]+(?=[A-ZÀ-ÖØ-ÝΑ-Я一-鿿])
      标点 + 空白 + 大写/CJK字符 → 候选句子边界

    Pass 2 — 缩写合并回补（核心创新）：
      - _ends_with_abbreviation() 检测前一段是否以已知缩写结尾
      - 5语言缩写词库（_ABBREV_PATTERNS_PER_LANG）：
        EN: Mrs|Mr|Prof|Dr|Inc|Ltd|Co|Corp|U.S|U.K|Ms|St|...
        FR: S.A.R.L|Mme|Mlle|S.A|E.U|Dr|Me|Pr|Ste|St|...
        DE: GmbH|Dr|Prof|Dipl|Ing|Hr|Fr|AG|KG|z.B|d.h|...
        IT: Sig.ra|S.p.A|S.r.l|Dott|Ing|Avv|Arch|Rag|Sig|...
        ES: S.A.de|Srta|S.A|S.L|Sra|Dr|Dra|Prof|Lic|Sr|...
      - 关键：长缩写排在短缩写之前（S.A.R.L 在 S.A 之前，
        Ste 在 St 之前），防止短前缀提前匹配
      - 递归合并：while-loop 持续吸收后续部分，直到当前段
        不再以缩写结尾。处理"连锁缩写"场景：
        "M. Dupont S.A." → 合并 "M." + "Dupont" + "S.A." 共3段

  ✅ 句子级 chunk 组装：
    - 贪婪地按句子分组，每个 chunk ≤ chunk_size tokens
    - 句子绝不从中切断 — 要么整句放进当前 chunk，要么开启新 chunk
    - Overlap 在句子级别：从上一个 chunk 末尾取尽可能多的完整句子
      填入 chunk_overlap tokens
    - 单句超长边缘情况：回退到 RecursiveCharacterTextSplitter，
      最强分隔符为 ". "（句号+空格），优先在从句边界切割

  实测效果（8672 chunks × 5语言）：
    法语 5313 chunks — S.A. / S.A.R.L. / Mme 等缩写正确处理
    英语 1408 chunks — Mr. / Ltd. / Inc. / U.S. 等缩写正确处理
    德语  685 chunks — GmbH / Dr. / z.B. / d.h. 等缩写正确处理
    意大利 440 chunks — S.p.A. / S.r.l. / Dott. 等缩写正确处理
    西班牙 288 chunks — S.A. / Sra. / Dr. 等缩写正确处理
    混合语言文档也正常处理（如 fr/en 309 chunks, fr/de 35 chunks）
